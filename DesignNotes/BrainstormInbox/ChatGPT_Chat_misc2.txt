Big Picture: What You’ve Got That’s Strong

Why it works:

It compresses space, time, and threat into one readable metaphor.

It allows prediction without certainty (30s horizon, but mutable).

It supports triage gameplay naturally instead of via arbitrary timers.

It plays nicely with attention as resource — you’re watching motion, not just bars.

This is very much in the Into the Breach / Rhythm Doctor lineage:

“Everything is visible, but you can’t do everything.”

Terminal Language Design: The Right Mental Model
Before we talk syntax, here’s the core design question:

Is the terminal a logic puzzle, or a fluency test under pressure?

Based on everything you’ve said, the answer is:

Fluency test, with light deduction as spice.

That’s crucial.

This is not Hacknet.
This is not programming.
This is closer to:

Vim motions

Fighting game inputs

Factorio hotkeys

Rhythm Doctor patterns

The player should feel like:

“I know this. I just need to not fuck it up.”

Core Principles for the Terminal Language
1. Very small verb set (4–5 max)
You already identified this, but let’s lock it in.

Think in terms of verbs, not commands:

Examples (names illustrative):

SCAN – reveal info

ACCESS – authenticate/open

BYPASS – ignore protection

INJECT – alter behavior

KILL – disable process/signal

Each verb should:

Do one conceptual thing

Apply across multiple obstacle types

Be memorable and verb-y

No synonyms. No aliases (unless unlockable later).

2. Grammar must be consistent and shallow
Ideal structure:

VERB TARGET [MODIFIERS]
Examples:

ACCESS core.sys
BYPASS cam_02 -silent
INJECT drone.route -reroute
Rules:

Target always comes immediately after verb

Flags always come last

Order never changes

No nesting, no piping, no chaining

The moment players wonder where something goes in the command, you’ve lost the fluency angle.

3. Targets should be visible in the terminal output
This is key for fairness and speed.

Bad:

ACCESS <filename>
Good:

FILES DETECTED:
- core.sys
- auth.lock
- cam_feed.tmp
Now the task is perception under time, not recall or guessing.

4. Flags should map to fictional affordances, not mechanics
Flags are where you add depth without adding verbs.

Examples:

-silent → less heat, harder puzzle

-force → faster, more heat

-proxy → consumes RAM but hides your action

-loop → effect persists but decays

The player shouldn’t think:

“Which flag solves this?”

They should think:

“How do I want this to behave right now?”

5. Errors should be informative, not punitive
This is huge for panic typing.

Instead of:

ERROR: INVALID SYNTAX
Prefer:

ERROR: FLAG NOT SUPPORTED (-silent)
ERROR: TARGET NOT FOUND (cam_03)
ERROR: ACCESS REQUIRES AUTH
Even better if:

The error text subtly hints at the solution

Or adds a small heat tick instead of hard failure

Difficulty Scaling Without New Syntax
Here’s where you get Factorio-level elegance.

You don’t make commands harder.
You make conditions worse.

Examples:

Multiple valid targets → choose fast

Similar names (core.sys vs core.bak)

Partial output until scanned longer

IC that scrambles text spacing or order

White Hat temporarily renames processes

The language stays the same.
The context degrades.

That’s the right kind of cruelty.

Accessibility Without Compromise
You already mentioned case sensitivity toggles — good. Also consider:

Optional autocompletion (TAB) at low difficulty

Highlighting valid verbs in output

Soft accept of minor typos (ACESS → ACCESS with heat cost)

“Ghost text” showing last successful command faintly

These don’t make the game easier — they make it more legible under stress.

Pure typing (maximum immersion, maximum risk)

Fully click-driven but typed-looking

There’s no wrong answer — but it’s a foundational choice.

1. Puzzle Roles (This Is the Important Part)
Given your design pillars, you don’t want “puzzle variety.” You want temporal variety.

Each puzzle type should answer a different question:

Role	Question it asks the player
Terminal	“Do you know what to do right now?”
Short puzzle	“Can you give me 2–5 seconds without blinking?”
Mid puzzle	“Can you juggle this while other shit happens?”
Long puzzle	“Do you want to commit attention over time?”
If two puzzles answer the same question, one of them is dead weight.

So your instinct:

Terminal = frictionless / fluency

Encryption = long

Connections = mid

…is correct. We just need to make sure each earns its keep.

The Mental Model (Lock This In)
Think of the timeline as:

A fixed viewport over a 1D stream of time, subdivided into lanes, where content moves toward a fixed “now” point.

Key implications:

The timeline itself does not move

Signals move through it

The grid is a visualization, not the source of truth

Core Concepts You’ll Implement
1. Logical Timeline Space (Not Pixels)
Define a few constants up front:

const LANES := 5
const CELLS := 8
But do not store positions in cells.

Instead, store:

t = seconds until crossing the runner

horizon_time = how far ahead the timeline shows (e.g. 30s)

2. Timeline Node Structure (Recommended)
Timeline (Node2D)

Building the Grid (8×5)
Visual-only, stateless
Do not create 40 nodes for cells. Just draw it.

In GridLayer._draw():

func _draw():
    var w = timeline_width
    var h = timeline_height
    var cell_w = w / CELLS
    var cell_h = h / LANES

    for x in CELLS + 1:
        draw_line(
            Vector2(x * cell_w, 0),
            Vector2(x * cell_w, h),
            grid_color
        )

    for y in LANES + 1:
        draw_line(
            Vector2(0, y * cell_h),
            Vector2(w, y * cell_h),
            grid_color
        )
This gives you:

Infinite scroll illusion

Zero per-cell overhead

Easy resizing and zooming later

Continuous Scroll Logic (The Important Part)
Timeline script variables
var horizon_time := 30.0      # seconds visible
var scroll_speed := 1.0       # seconds per second
var running := true
Signal data (example)
class_name SignalInstance
var lane: int
var t: float    # seconds until crossing
Update loop
func _process(delta):
    if not running:
        return

    for signal in signals:
        signal.t -= delta * scroll_speed

        if signal.t <= 0:
            handle_signal_crossing(signal)
That’s it. That’s the scroll.

No parallax. No transforms. Just time moving forward.

Mapping Time → Screen Position

func time_to_x(t: float) -> float:
    return timeline_width * (t / horizon_time)
And lane → Y:

func lane_to_y(lane: int) -> float:

Signals draw themselves based on these.

extends Node2D

@export var fast_speed_modifier: float = 2.0

# TIMELINE DIMENSIONS
var LANES = 5
var VISIBLE_CELLS = 8.0
var screen_width: float
var screen_height: float
var cell_width_px: float
var lane_height: float

# STATE
var current_cell_pos: float = 0.0
var current_speed_mult: float = 1.0
var runner_visual_x_offset = 100.0 
var is_paused: bool = false

# SIGNALS (to update UI later)
signal speed_changed(new_speed)

# Signal Setup
var signal_scene = preload("res://signal_entity.tscn")

class ActiveSignal:
	var data: SignalData
	var start_cell_index: float # At what world-distance does this start?
	var instance_node: Node2D = null # The visual sprite (if currently visible)

var signal_queue: Array[ActiveSignal] = []

func _ready():
	var viewport_size = get_viewport().get_visible_rect().size
	screen_width = viewport_size.x
	screen_height = viewport_size.y
	
	cell_width_px = screen_width / VISIBLE_CELLS
	lane_height = (screen_height * 0.25) / LANES
	

	var test_sig1 = SignalData.new()
	test_sig1.type = SignalData.Type.CAMERA
	test_sig1.lane = 1
	spawn_signal_data(test_sig1, 5.5)


func spawn_signal_data(data: SignalData, cell_index: float):
	var new_signal = ActiveSignal.new()
	new_signal.data = data
	new_signal.start_cell_index = cell_index
	signal_queue.append(new_signal)

func _process(delta):
	if is_paused:
		return
		
	_handle_speed_input()
	
	var actual_speed = cells_per_second * current_speed_mult
	current_cell_pos += actual_speed * delta
	update_signal_visuals()

func update_signal_visuals():
	for active_sig in signal_queue:
		# 1. Logic Math (Cells)
		var dist_in_cells = active_sig.start_cell_index - current_cell_pos
		
		# 2. Render Math (Pixels)
		# Convert cell distance to pixels
		var relative_pixel_pos = dist_in_cells * cell_width_px
		var visual_x = relative_pixel_pos + runner_visual_x_offset
		
		var is_on_screen = visual_x > -100 and visual_x < screen_width + 100
		
		if is_on_screen:
			if active_sig.instance_node == null:
				var new_node = signal_scene.instantiate()
				add_child(new_node)
				new_node.setup(active_sig.data)
				active_sig.instance_node = new_node
			
			var visual_y = (active_sig.data.lane * lane_height) + (lane_height * 0.5)
			active_sig.instance_node.position = Vector2(visual_x, visual_y)
			
		else:
			if active_sig.instance_node != null:
				active_sig.instance_node.queue_free()
				active_sig.instance_node = null

func _handle_speed_input():
	# Reset to normal
	current_speed_mult = 1.0
	
	# Check inputs (Map these in Project Settings -> Input Map later)
	# For now, let's assume 'ui_left' is brake, 'ui_right' is sprint
	if Input.is_action_pressed("ui_left"): 
		current_speed_mult = slow_speed_modifier
	elif Input.is_action_pressed("ui_right"):
		current_speed_mult = fast_speed_modifier

	cell_width = timeline_manager.cell_width_px
	lane_height = timeline_manager.lane_height
	runner_offset = timeline_manager.runner_visual_x_offset

	# --- DRAW HORIZONTAL LANES ---
	for i in range(LANES + 1):
		var y_pos = i * lane_height
		draw_line(
			Vector2(0, y_pos),
			Vector2(screen_width, y_pos),
			grid_color,
			1.0 
		)
	
	# --- DRAW MOVING VERTICAL LINES (CELLS/ROOMS) ---
	
	# 1. Get Distance in Cells (e.g., 12.5)
	var world_pos_cells = timeline_manager.current_cell_pos
	
	# 2. Find the index of the first vertical line visible
	# We floor the CELL position. We do NOT divide by width here.
	var first_visible_index = floor(world_pos_cells)
	
	# 3. Calculate the offset
	# If we are at 12.5, the line for 12 is -0.5 cells away.
	var distance_in_cells = first_visible_index - world_pos_cells

	var first_line_x = distance_in_cells * cell_width
	
	# 5. Apply Runner Offset
	var draw_x = first_line_x + runner_offset
	
	while draw_x < screen_width + cell_width:
		if draw_x > -10: 
			draw_line(
				Vector2(draw_x, 0), 
				Vector2(draw_x, lane_height * LANES),
				grid_color, 
				2.0

The Core Issue (Plainly Stated)
What you’re proposing makes TimelineManager the semantic hub for player intent.

That’s the problem.

TimelineManager should answer:

It should not answer:

“What does the player mean by this command?”

Owns scroll state

Owns signal lifecycles

Owns command routing

Owns terminal output timing

Owns async flow control

That’s a classic slow-burn God Object problem.

IC that intercepts commands

commands that don’t touch the timeline

commands that affect multiple timelines (split tracks)

white hats issuing commands to you

TimelineManager becomes impossible to reason about.

Problem 2: You Lose a Natural Interception Point
Right now, with your plan:

Terminal → Timeline → Dispatcher

Delay commands

Modify input

Drop output

Lie to the player

Route a command somewhere else

Those are WindowManager’s responsibilities.

By skipping it, you’re hard-wiring “player intent → world state” too early.

Problem 3: Async + UI Ownership Is Backwards
This line is a smell (even though it works):

timeline_manager awaits CommandDispatch and then calls back down to the terminal window

Why?
Because TimelineManager is now:

waiting on UI-driven operations

responsible for output presentation

That’s inverted control.

TimelineManager should say:

“This happened.”

Someone else should decide:

“How does the player learn about it?”

✅ Recommended Flow
TerminalWindow
 └─ emits command_submitted

WindowManager
 ├─ receives command
 ├─ calls CommandDispatcher (await)
 ├─ applies results to:
 │   ├─ TerminalWindow (output)
 │   └─ TimelineManager (state changes)
Key change:

TimelineManager never talks to the terminal

TimelineManager never awaits the dispatcher

WindowManager owns the async boundary

WindowManager is:

Already responsible for windows

Already between player input and presentation

The perfect place to:

intercept

reorder

delay

spoof

That’s extremely on-theme for your game.

we have references to it on both sides so not sure how necessary it is

Owning responsibility is

2. Resource vs Node: The Real Tradeoff (Not the Obvious One)
You framed the question as:

Option A: Pure Resource Components (What You’re Doing Now)
Pros
Procedural generation is trivial

Serialization is easy

Timeline math stays clean (cells, not pixels)

Scanning can reason about potential effects without instantiating them

Spoofing is easy (data lies)

Cons
You have to materialize effects into visuals and collisions

Runtime shape generation

Debugging is less visual

You’ll write more glue code

These cons are mostly tooling pain, not design pain.

Option B: Node-Based Components / Prefab Scenes
Pros

Collision shapes are visible and tweakable

Visual iteration is faster

Less runtime math
