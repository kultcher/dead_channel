Have a couple different paradigms I'm thinking about, which may work complimentarily:

Passive programs with small benefits and a consistent RAM cost, you may have more programs than RAM, so juggling your programs vs. RAM becomes part of the loop
Active, instant or short-duration programs with a "load and/or cleanup" process that occupies RAM for a duration (i.e. activate a program, it does the thing, then occupies 2 RAM while it runs cleanup for 5 seconds OR alternately, a program that occupies RAM during a 5s load period, and it can't be used until it loads so you have to prep for it.
As for what programs can do, I have several broad design spaces:

Puzzle helpers: Make minigames easier
Passive example: Speeds up Decrypt cull cycles
Active example: Completely solves one character instantly

Terminal overrides: Make your terminal commands function better
Passive example: Highlights useful terminal info to seperate it from noise
Active example: Your next KILL doesn't generate any heat

Anti IC: Helps against specific types of IC (maybe too narrow, but can be cheap or have themed packages)
Passive example: Trace IC takes longer
Active example: Purges a random IC module from a signal

Runner Support: Affects the runner team
Passive example: Gives runners bonuses on their rolls (or whatever system we use) against obstacles
Active example: Might be tough to do discrete bonuses at this scale without breaking the game (i.e. healing the runners would be overpowered if you could do it more than once or twice in a run)

Heat Management:
Passive example: Reduces all heat gain by a %
Active example: Dumps large amount of heat but has a long cooldown

Scanning:
Passive example: Each scan reveals 2 layers
Active example: Instantly fully scan all signals (or signals within a range of selected signal)

Metagame:

Passive example: Increases value of paydata

Active example: Haven't thought through metagame systems enough to think of any
