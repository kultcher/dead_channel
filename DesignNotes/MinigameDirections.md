# Dead Channel Minigame Directions

This file is a compiled idea bank for minigame behavior, modifiers, and future expansion directions. It is not a commitment to build everything listed here.

Current prototype roster:

- Terminal
- Sniff
- Decrypt
- Fuzz

## Governing Principles

- Minigames should reinforce the wider run fantasy, not hijack it.
- Distinct minigames should justify their existence through different time profiles and attention demands.
- More variety should usually come from modifiers, IC, and context before adding whole new puzzle families.
- The puzzle should rarely be the only source of difficulty; the surrounding run state should matter.

## Role Map

### Terminal

Role:

- recurring command / interpretation layer
- broadest and most central "micro-puzzle"

Expansion directions:

- more nuanced process/property targeting
- better interaction with signal state
- richer command-context feedback

### Sniff

Role:

- fast attention check
- short eyes-on-screen task

Strengths:

- immediate pressure
- low explanation burden
- easy to fit into the wider triage loop

Expansion directions:

- density / speed scaling
- clearer integration with scan quality and hostile noise
- light variants that change readability without changing the core verb

Modification ideas:

- decoys
- shifting target signatures
- partial signal corruption
- short-lived windows of clarity

### Decrypt

Role:

- slow background process
- periodic check-in puzzle
- deduction plus wait-state tension

Strengths:

- creates a plate the player keeps spinning
- supports background progress while other threats unfold

Expansion directions:

- richer partial-deduction rewards
- better focus mechanics
- stronger interaction with heat and IC

Modification ideas:

- manual candidate culling
- momentum boosts from correct slots
- focus decay
- linked keyspaces
- misleading or rekeying transforms

### Fuzz

Role:

- timing / probing / stacking-success prototype
- candidate for a more active "pressure rhythm" puzzle without simply being a rhythm minigame

What it should prove:

- whether repeated small successful reads feel distinct from Sniff and Decrypt
- whether timing windows and probing create good tension under multitasking pressure
- whether it produces a useful middle-speed puzzle profile

Potential directions:

- probing weak points to reveal exploitable windows
- stacking successful hits to build a breakthrough state
- trading patience for accuracy
- escalating instability if the player pushes too hard

## Cross-Puzzle Modifier Bank

These are useful ways to vary puzzles without inventing brand-new systems.

### Information modifiers

- reduced clarity
- decoys
- incomplete data
- delayed reveal
- misleading labels

### Pace modifiers

- shorter solve windows
- passive decay
- burst phases
- delayed resolution
- sudden state resets

### Attention modifiers

- puzzle asks for periodic re-engagement
- puzzle competes with other high-priority windows
- puzzle output affects what the player can see elsewhere

### Input modifiers

- temporary inversion or reordering
- narrowed valid interaction window
- partial lockouts after mistakes

### State modifiers

- solved sections can destabilize
- progress can cascade
- puzzle can be partially completed and left

## IC-Friendly Puzzle Hooks

These are especially useful because they let puzzles vary through hostile system behavior.

- rekeying
- noise injection
- fake solved states
- blocked autofill
- linked puzzle elements
- encoded candidates
- delayed confirmation
- scan corruption

## Future Puzzle Families

These are explicitly exploratory. They should only move forward if the current roster leaves a real gap.

- spatial routing / connection puzzles
- layered-information puzzles
- memory-heavy variants
- rhythm-adjacent timing variants
- twisting / sustained-maintenance tasks

## Filters For New Puzzle Ideas

Before adding a new puzzle family, ask:

- Does it have a distinct time profile?
- Does it create a different kind of attention pressure?
- Does it fit the hacker-support fantasy?
- Could the desired variety be achieved with modifiers instead?
- Is it teachable without bloating the tutorial burden?

If the answer to the last two questions is bad, it probably belongs in the modifier bank instead.
